# 사용자 고민 → 기능 요구사항 매핑

> 작성일: 2026-04-08
> 목적: 사용자가 Claude Code를 사용하면서 느낀 고민을 구체적인 기능 요구사항으로 변환

---

## 원문 고민

### 고민 1. 작업 문서 설계의 중요성
> 클로드코드를 잘 사용하기 위해서는 클로드코드가 스스로 지속적으로 관리하는 작업문서 설계가 중요하다.
> ex: 요구사항 수집문서, 기획서, PRD, 작업현황 문서, 세션히스토리 기록문서, 로드맵, 마일스톤 등

### 고민 2. 프롬프트 대화의 태스크화
> 사용자는 로컬 터미널에서 프롬프트 대화로 클로드코드에게 질문, 지시를 지속적으로 반복하는데, 구체적으로 해당 작업이 task화 되어 관리 되었으면 좋겠다.

### 고민 3. 태스크 워크플로우
> 이 태스크는 사용자가 셋팅한 워크플로우대로 task가 지속적으로 update되고 진행단계를 거쳐 최종 완료 or 실패 or 재시도 등이 되어야 한다.

### 고민 4. 실시간 모니터링
> 클로드 코드의 실시간 모니터링을 하고 싶다.

### 고민 5. 문서 관리 감독
> 클로드코드가 작성하는 문서들은 너무 방대하여 지속적으로 관리 감독이 필요하고, 중요도를 분류하여 중요한 내용은 사용자가 지속적으로 보고 검토할 수 있어야 한다.

---

## 고민별 분석 및 기능 요구사항

### 고민 1 → 문서 라이프사이클 관리

**현재 상태**: Claude Code는 CLAUDE.md, MEMORY.md, rules 등으로 문서를 관리하지만, **어떤 문서가 있고 언제 갱신되었는지 모니터링하는 도구가 없다**.

**필요한 기능**:

| ID | 기능명 | 설명 |
|----|--------|------|
| UC-1A | **문서 인벤토리 뷰** | 프로젝트 내 Claude Code 관리 문서 목록 표시 — CLAUDE.md, MEMORY.md, rules/, skills/, agents/, docs/ 하위 문서 등. 각 문서의 최종 수정일, 크기, 라인 수 표시 |
| UC-1B | **문서 변경 타임라인** | 세션별로 어떤 문서가 생성/수정/삭제되었는지 시간순 표시. file-history 데이터 활용 |
| UC-1C | **문서 건강 검사** | Readout의 Lint 기능 참고 — CLAUDE.md 구조 검사, MEMORY.md 크기 경고, 누락된 필수 문서 알림 |

**Readout 대응**: Lint(CLAUDE.md 린터), Memory(MEMORY.md 뷰어), Diffs(세션별 파일 변경)
**현재 PRD 대응**: F6(파일 변경 하이라이트) — 부분적 커버

---

### 고민 2 → 프롬프트 → 태스크 자동 추출

**현재 상태**: Claude Code는 내부적으로 TaskCreate/TaskUpdate 도구로 태스크를 관리하지만, `~/.claude/todos/` 에 저장된 태스크를 **외부에서 조회/관리하는 방법이 없다**.

**필요한 기능**:

| ID | 기능명 | 설명 |
|----|--------|------|
| UC-2A | **태스크 보드 뷰** | `~/.claude/todos/{sessionId}-*.json` 파일을 파싱하여 칸반 보드 형태로 표시 (pending → in_progress → completed/deleted) |
| UC-2B | **프롬프트 → 태스크 매핑** | 사용자의 각 프롬프트(user 레코드)가 어떤 태스크를 생성했는지 연결. JSONL의 TaskCreate tool_use와 user 레코드의 parentUuid 체인으로 추적 |
| UC-2C | **태스크 히스토리** | 태스크의 상태 변경 이력 표시 (생성 → 진행중 → 완료). TaskUpdate tool_use의 timestamp 기반 |

**데이터 소스**:
```
~/.claude/todos/{sessionId}-agent-{sessionId}.json
  → 태스크 목록 (subject, description, status, owner 등)

JSONL에서 tool_use: TaskCreate, TaskUpdate
  → 태스크 생성/변경 이벤트
```

**Readout 대응**: 직접 대응 기능 없음 — **zm-agent-manager만의 차별화 기능**
**현재 PRD 대응**: 없음 — **신규 기능 제안**

---

### 고민 3 → 사용자 정의 워크플로우

**현재 상태**: Claude Code의 태스크는 단순 상태(pending/in_progress/completed/deleted)만 가짐. 사용자가 정의한 **커스텀 워크플로우**(예: 기획→설계→구현→리뷰→완료)로 관리할 방법이 없다.

**필요한 기능**:

| ID | 기능명 | 설명 |
|----|--------|------|
| UC-3A | **워크플로우 정의** | 사용자가 태스크 진행 단계를 정의 (예: `요구사항 수집 → 설계 → 구현 → 테스트 → 리뷰 → 완료`). 각 단계에 조건(자동 전환) 설정 가능 |
| UC-3B | **태스크 상태 매핑** | Claude Code의 내장 상태(pending/in_progress/completed)를 사용자 워크플로우 단계에 매핑 |
| UC-3C | **자동 상태 전환** | 특정 조건 충족 시 태스크 단계 자동 전환 (예: 코드 커밋 시 "구현"→"리뷰" 자동 이동) |
| UC-3D | **실패/재시도 처리** | 태스크 실패 시 이전 단계로 되돌리기, 재시도 카운트, 실패 사유 기록 |

**Readout 대응**: 없음 — **zm-agent-manager만의 핵심 차별화 기능**
**현재 PRD 대응**: 없음 — **신규 기능 제안**

---

### 고민 4 → 실시간 모니터링

**현재 상태**: 이미 PRD에 F1~F4로 정의됨. Readout의 Live/Sessions/Tools와 대응.

**필요한 기능**:

| ID | 기능명 | 설명 | PRD 매핑 |
|----|--------|------|---------|
| UC-4A | **실시간 세션 감시** | 활성 세션 실시간 표시 (상태, 경과 시간, 메모리) | F1, F2 |
| UC-4B | **도구 호출 실시간 피드** | 현재 실행 중인 도구 호출을 실시간으로 표시 | F4 |
| UC-4C | **메시지 스트리밍** | 새 user/assistant 메시지를 실시간으로 타임라인에 추가 | F2, F3 |
| UC-4D | **비용 실시간 추적** | 현재 세션의 누적 비용을 실시간 표시 | (신규, Readout Costs 참고) |
| UC-4E | **알림** | 세션 시작/종료, 에이전트 stuck, 비용 임계치 도달 시 데스크톱 알림 | (신규, INBOX #5) |

**Readout 대응**: Live, Sessions, Tools, Costs, Dashboard
**현재 PRD 대응**: F1~F4 (부분적), F8 (통계)

---

### 고민 5 → 문서 중요도 분류 및 감독

**현재 상태**: Claude Code가 생성/수정하는 문서를 **중요도별로 분류하고 사용자에게 리뷰를 요청하는 메커니즘이 없다**.

**필요한 기능**:

| ID | 기능명 | 설명 |
|----|--------|------|
| UC-5A | **문서 변경 알림 (중요도별)** | 세션 중 변경된 문서를 중요도별로 분류하여 알림. 중요도 기준: PRD/요구사항 변경 > 정책 변경 > 로드맵 변경 > 일반 문서 변경 |
| UC-5B | **문서 리뷰 큐** | 사용자가 리뷰해야 할 문서 변경 목록을 큐 형태로 관리. 각 변경에 대해 "승인" / "반려" / "수정 요청" 액션 |
| UC-5C | **문서 diff 뷰** | 변경된 문서의 이전/이후 버전을 side-by-side diff로 표시. file-history 데이터 활용 |
| UC-5D | **문서 대시보드** | 모든 관리 문서의 현재 상태, 최종 수정 시간, 크기, 변경 빈도를 한눈에 표시 |

**Readout 대응**: Diffs(파일 변경), Lint(문서 건강), Repo Pulse(미커밋 파일)
**현재 PRD 대응**: F6(파일 변경 하이라이트) — 부분적 커버

## Readout과의 차별화 방향

```
Readout (현재)
  └─ "개발 환경을 한눈에 보는 대시보드"
  └─ git/ports/deps/env 등 범용 개발 도구 통합
  └─ 세션 리플레이 + 비용 추적

zm-agent-manager (제안)
  └─ "Claude Code 작업을 관리하는 프로젝트 매니저"
  └─ 태스크 보드 + 워크플로우 (Readout에 없는 핵심 차별화)
  └─ 문서 라이프사이클 관리 + 리뷰 큐 (Readout에 없는 핵심 차별화)
  └─ 실시간 모니터링 + 세션 리플레이 (Readout과 동등)
  └─ 비용 추적 + 알림 (Readout과 동등)
```

**핵심 포지셔닝**: Readout이 **"모니터"** 라면, zm-agent-manager는 **"매니저"** 다.

## todos 데이터 구조 (태스크 보드 구현을 위한 사전 조사 필요)

```
~/.claude/todos/{sessionId}-agent-{sessionId}.json
```

> 이 파일의 실제 구조를 분석하여 태스크 보드 기능의 데이터 소스를 확정해야 함.
> 분석 대상: 필드 구조, 상태값, 생성/수정 시간, 연관 세션 ID 등.
