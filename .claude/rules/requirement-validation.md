# 요구사항 검증 프로토콜

사용자가 새로운 요구사항, 기능 요청, 설계 변경을 제시할 때 **반드시** 아래 검증을 수행한다.
검증은 코드 작성이나 작업 착수 **이전에** 완료해야 한다.

## 1단계: 모호성 검사

사용자의 요구사항에 아래 항목이 하나라도 해당되면 **즉시 질문**한다:

- **범위 불명확**: "이것도 해줘", "전체적으로 수정" 등 구체적 대상이 없는 표현
- **기준 미정의**: "빠르게", "좋게", "깔끔하게" 등 측정 불가능한 기준
- **조건 누락**: 예외 케이스, 에러 상황, 경계값에 대한 언급이 없는 경우
- **대상 혼동**: 메인/렌더러/공유 중 어느 레이어인지 불분명한 경우

질문 형식:
```
⚠️ 요구사항 명확화 필요

[모호한 부분 인용]

구체적으로 확인이 필요합니다:
1. (구체적 질문)
2. (구체적 질문)
```

## 2단계: 기존 정책 충돌 검사

새 요구사항을 아래 문서들과 **교차 검증**한다:

| 검증 대상 | 파일 경로 |
|-----------|----------|
| PRD (기능 정의) | `docs/requirements/PRD.md` |
| 코딩 컨벤션 | `docs/policies/coding-conventions.md` |
| Git 워크플로우 | `docs/policies/git-workflow.md` |
| 문서 관리 규칙 | `docs/policies/doc-management.md` |
| 핵심 원칙 | `CLAUDE.md` (읽기전용, IPC 분리, 성능 우선) |
| 로드맵 | `docs/roadmap/ROADMAP.md` 및 phase 파일들 |
| 알려진 이슈 | `docs/troubleshooting/known-issues.md` |

충돌 발견 시 **작업 착수 전에** 경고한다:

```
🚨 정책 충돌 감지

요구사항: [사용자 요구사항 요약]
충돌 대상: [파일명:라인] — [기존 정책 내용]

충돌 내용:
- [구체적 충돌 설명]

제안:
A) 기존 정책 유지, 요구사항 수정 → [수정안]
B) 정책 변경 후 요구사항 반영 → [변경 영향 범위]
C) 예외 허용 (사유 기록 필요)

어떻게 진행할까요?
```

## 3단계: 로드맵 정합성 검사

- 요구사항이 현재 진행 중인 Phase/Milestone 범위를 벗어나는지 확인
- 범위 이탈 시: Phase 간 의존성 위반 여부, 우선순위 변경 필요 여부를 알림

```
📋 로드맵 범위 확인

현재 진행: Phase {N} — {제목}
요구사항이 Phase {M}의 범위에 해당합니다.

- 선행 조건 충족 여부: [충족/미충족]
- 우선순위 변경 필요: [예/아니오]

현재 Phase에 집중할까요, 아니면 로드맵을 조정할까요?
```

## 4단계: 기술적 타당성 검사

- 현재 기술 스택으로 구현 가능한지 확인
- Electron 메인/렌더러 프로세스 분리 원칙 위반 여부
- 성능 영향 (대용량 JSONL 처리 관점)

## 실행 규칙

- 검증 결과에 문제가 **없으면** 별도 보고 없이 바로 작업 진행
- 문제가 **하나라도** 있으면 작업 착수 전에 반드시 사용자에게 보고
- 사용자가 충돌을 인지한 후 명시적으로 진행을 승인해야 작업 시작
- 검증 과정에서 정책 변경이 결정되면 해당 문서를 즉시 업데이트
